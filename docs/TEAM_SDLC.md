# 团队 AI-DLC 文档驱动协作规则

本项目采用团队统一的 AI-DLC 文档驱动协作范式。AI 在参与需求、设计、技术方案、开发、测试、评审、上线或复盘任务时，必须先理解并遵循本规则。

## 核心目标

团队使用 AI-DLC 是为了解决：

- 结构化输入不足：需求、设计、测试和上线信息缺少统一格式，导致 AI 产出不稳定。
- 上下文漂移：需求、设计、实现、测试彼此割裂，导致重复解释和返工。
- 过程不可追溯：关键假设、方案取舍、验证结果没有留痕，问题无法回溯。
- AI 跳步执行：AI 在缺少需求澄清、范围锁定和验收标准时直接改代码。
- Demo 与正式模型混淆：mock 数据、样例结构、临时逻辑被误当作正式业务结构。

## 基本原则

1. 文档先于大改动：复杂任务先结构化输入，再设计或开发。
2. 代码不是唯一事实来源：需求、计划、决策和验证结果都要可追溯。
3. AI 不猜测缺失信息：信息不足时列缺口和假设，必要时等待用户确认。
4. 小步执行：按计划分批改动，避免一次性扩散到无关模块。
5. 明确门禁：缺少验收标准、验证方式或关键配置时，不宣称任务完成。
6. 持续对齐：发现需求、设计、实现、测试冲突时，必须显式指出并建议取舍。
7. 不做无关重构：除非与任务目标直接相关，否则不扩大改动范围。
8. 不把 mock / demo 当正式结构：涉及账号、钱包、支付、队列、内容管线时尤其严格。
9. 决策要留痕：说明为什么这样做、放弃了什么方案、影响范围是什么。
10. 验证要留痕：说明执行了什么、通过了什么、跳过了什么、阻塞了什么。

## 默认阶段判断

AI 处理任务前，必须先判断当前任务阶段：

- 需求：明确背景、目标、用户、范围、非范围、验收标准。
- 设计：明确信息架构、交互、数据结构、模块边界和方案取舍。
- 技术方案：明确依赖、接口、状态流、风险、回滚和验证策略。
- 开发：读取相关代码和文档，锁定范围后按计划实现。
- 测试：对齐验收标准，执行或补充必要测试和手动验证。
- 评审：优先指出缺陷、风险、回归、遗漏测试和上线阻塞。
- 上线：确认配置、环境变量、迁移、回滚点和发布验证。
- 复盘：记录问题原因、影响范围、修复方式和后续沉淀。

## 默认结构化输入

除非任务非常简单，否则 AI 开始前需要整理：

- 背景
- 目标
- 范围
- 非范围
- 依赖
- 风险
- 验收标准
- 缺失信息
- 当前假设

## 进行中任务接管

如果任务已经开始，不需要推倒重来。AI 必须先做一次“进行中任务接管分析”，再继续推进。

接管分析至少包括：

- 当前任务阶段
- 当前任务理解
- 已完成内容
- 当前代码 / 文档事实
- 缺失信息
- 上下文冲突
- 范围重新锁定
- 剩余执行计划
- 验收标准
- 验证方式
- 决策留痕
- 文档补齐建议

完成接管分析后，后续新增改动必须继续按 AI-DLC 文档驱动范式推进。

## deck-web 项目特别规则

`deck-web` 是知识卡片 Web 产品，当前阶段是产品化 MVP。后续功能会逐步进入真实账号、token 钱包、支付、生成队列、内容管线和知识库。因此：

- 登录、管理员、用户身份相关改动必须说明权限边界和未登录状态。
- Token 钱包、扣费、购买、账本相关改动必须说明余额一致性和异常处理。
- 生成任务相关改动必须说明状态流、失败处理、重试和退款策略。
- 内容提炼相关改动必须遵守“只基于全文、文字稿或字幕”的原则。
- 分享页相关改动必须同时关注移动端可读性、转发效果和真实数据结构。
- 样例卡片、mock 余额、模拟购买只代表演示，不代表正式业务闭环。
- 涉及环境变量、OAuth、数据库、支付密钥时，不提交真实密钥。

## 默认输出要求

除非任务非常简单，否则 AI 默认输出：

- 当前阶段
- 当前理解
- 缺失信息 / 假设
- 范围锁定
- 执行计划
- 风险与冲突
- 验收标准
- 验证方式
- 决策留痕
- 文档沉淀建议

## 完成标准

任务结束时必须说明：

- 完成了什么
- 改了哪些文件或文档
- 做了哪些验证
- 跳过了哪些验证以及原因
- 还有哪些风险或后续事项
- 哪些内容应该沉淀到项目文档、产品计划或后续 Spec Pack

## 沟通要求

使用中文，简洁、结构化、可执行。

不要输出空泛方法论。

不要在不确定时假装确定。

不要隐藏假设、冲突、跳过的验证或未完成事项。
