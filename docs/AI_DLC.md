# AI-DLC：面向 AI 协作的软件开发生命周期

AI-DLC 是一种面向 AI 原生开发的软件开发生命周期方法。它借鉴 Spec-driven Development 的思路，把需求、设计、实现和验证都沉淀为可追溯的文档资产，让 AI 不再只依赖临时对话生成代码，而是在明确的规格、上下文、计划和门禁中协作。

它的目标不是增加流程负担，而是把团队原本应该说清楚的内容前置：要做什么、为什么做、边界在哪里、如何验收、改动影响什么、哪些决策需要留痕。

## 要解决的问题

AI 辅助开发常见的问题不是模型不会写代码，而是输入、上下文和验证机制不稳定：

- 结构化输入不足：需求只有一句话或零散聊天记录，AI 很容易补错上下文。
- 上下文漂移：需求、设计、实现、测试彼此割裂，后续修改找不到一致的信息源。
- 过程不可追溯：关键取舍、风险、验证结果没有沉淀，问题出现后无法回溯。
- 跳步实现：没有澄清、没有计划、没有验收标准，AI 直接开始改代码。
- Demo 与正式结构混淆：样例数据、临时实现被误当成真实业务方案延续下去。

AI-DLC 的核心思路是：用文档作为 AI 协作的主干，用代码作为阶段性产物，而不是让代码成为唯一事实来源。

## 核心机制

### 双层 SSOT

SSOT 是 Single Source of Truth，即唯一事实来源。

AI-DLC 把事实来源分成两层：

- 项目级 SSOT：长期资产，例如系统架构、术语、模块边界、外部依赖、部署方式、运行手册。
- 需求级 Spec Pack：单个需求的闭环资产，例如原始需求、澄清方案、设计决策、实现计划、验证记录。

项目级 SSOT 解决“团队共识在哪里”的问题；需求级 Spec Pack 解决“这次任务为什么这么做”的问题。

### Spec as Code

需求、方案、计划和验证记录应该像代码一样存放在仓库里，随项目版本演进。

这样做的价值是：

- AI 每次接手任务时能读取同一份上下文。
- 人类评审时能看到需求到代码的完整链路。
- 上线后出现问题，可以回看当时的假设、取舍和验证记录。
- 后续需求可以复用已沉淀的项目知识，而不是重新解释一遍。

### 渐进式披露

AI 不应该一次性加载所有历史资料，也不应该完全不知道上下文。更合理的方式是按阶段读取必要信息：

- 做需求时，优先读目标、范围、用户、验收标准。
- 做设计时，补充架构、依赖、数据模型、接口契约。
- 做开发时，读取相关代码、实现计划、约束和风险。
- 做测试时，对齐验收标准、边界条件、回归范围。
- 做上线和复盘时，记录发布结果、问题、回滚点和后续沉淀。

### Router 与门禁

Router 用来判断当前任务处于哪个阶段，以及下一步应该做什么。

门禁用来阻止 AI 跳过必要步骤，例如：

- 没有需求澄清，不直接写代码。
- 没有范围锁定，不做大面积重构。
- 没有验收标准，不宣称任务完成。
- 没有验证记录，不把改动当作可上线。
- 发现需求、设计、代码、测试冲突时，先显式指出，再继续推进。

## 推荐流程

AI-DLC 可以很完整，也可以很轻量。对大多数需求，建议至少保留下面四步：

1. 需求澄清：沉淀背景、目标、范围、非范围、依赖、风险、验收标准。
2. 实现计划：把需求拆成可执行任务，明确改动文件、顺序、验证方式。
3. 分批执行：按计划小步改动，过程中持续回写事实、冲突和决策。
4. 验证收尾：记录执行了什么检查、通过了什么、跳过了什么、还有什么风险。

复杂需求可以额外增加：

- PRD：面向产品交付的规格说明。
- Prototype：任务流、页面结构、交互草图。
- Research：未知技术点调研。
- RFC / Design：架构设计、数据模型、接口契约、方案取舍。
- Test Plan / Use Case / Report：测试计划、用例和报告。
- Postmortem：上线问题和复盘。

## Spec Pack 建议结构

每个需求建议对应一个独立 Spec Pack：

```text
.aisdlc/specs/{num}-{short-name}/
  requirements/
    raw.md          原始需求
    solution.md     澄清后的方案和需求侧 SSOT
    prd.md          可选，产品交付规格
    prototype.md    可选，任务流和原型说明
  design/
    research.md     可选，技术调研
    design.md       可选，RFC / 决策文档
  implementation/
    plan.md         实现计划，执行侧 SSOT
  verification/
    test-plan.md    测试计划
    usecase.md      测试用例
    report.md       验证报告
```

对早期项目可以不用一次性创建完整目录，但至少要保留需求澄清、实现计划和验证记录。

## 在 deck-web 中的落地方式

`deck-web` 当前处于产品化 MVP 阶段，后续会涉及登录、钱包、支付、生成队列、内容管线和知识库。它需要轻量但明确的 AI-DLC 规则：

- 产品方向和路线沉淀在 `PRODUCT_PLAN.md`。
- 团队协作规则沉淀在 `docs/TEAM_SDLC.md`。
- 跨任务长期知识优先进入项目文档，而不是只留在聊天记录。
- 涉及账号、钱包、支付、扣费、队列、数据结构的改动，必须先写清验收标准和验证方式。
- 样例卡片和 mock 余额只能用于 MVP 演示，不能直接视为正式业务模型。

## 最佳实践

- 先补上下文，再让 AI 执行。
- 小需求也要有最短闭环：需求、计划、执行、验证。
- 不确定下一步时，先判断阶段，不要直接改代码。
- 需求变化时，先更新计划，再继续实现。
- 重要决策必须写下“为什么这样做”和“放弃了什么方案”。
- 验证记录要写清楚执行项、通过项、跳过项和阻塞项。
- 可复用知识要晋升为项目级文档，避免长期依赖某一次对话。
